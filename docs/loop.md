# 스킬 강화 루프 — 운영 규약 (정본)

이 마켓플레이스의 운영 교리. **프로젝트를 반복할수록 스킬이 강해지는 루프**를 어떻게 도는지 정의한다.
이 문서가 규약의 단일 정본(SSOT)이다 — 다른 컴퓨터에서도 이 repo만 있으면 규약이 딸려온다.

허브(이 repo를 클론해 두고 승격 세션을 돌리는 작업 폴더)의 구조:

```
<hub>/
├─ skills-marketplace/   # 이 repo — 정본 + 배포 (github.com/sungwooHa/skills-marketplace)
└─ _capture/             # 프로젝트에서 수거한 원본 피드백 (승격 대기소, 미가공)
```

---

## 루프 (부품 4개)

| 부품 | 하는 일 | 어디서 |
|---|---|---|
| **정본** | 진실이 사는 곳 | 이 repo (`plugins/`) |
| **배포** | 프로젝트가 가져다 씀 | `/plugin install <name>@skills-marketplace` |
| **포착** | 마찰을 기록으로 전환 | 각 프로젝트 `.claude/skill-feedback.md` |
| **승격** | 기록을 검증해 정본에 병합 | `_capture/` → 승격 세션 → 정본 |

**핵심 비대칭: 포착은 공짜로, 승격은 비싸게.**
포착이 비싸면 아무것도 안 잡히고, 승격이 싸면 오염되고 비대해진다.

---

## 1. 포착 (프로젝트 작업 중 — 판단하지 않는다)

스킬이 기대만큼 못 할 때, 그 프로젝트의 `.claude/skill-feedback.md`에 **한 줄 append**.

```markdown
- [스킬명] 무슨 마찰 — 최소 재현 예시 / 기대 vs 실제
- [루프/<부품>] 루프 자체의 마찰 — 부품 = 정본·배포·포착·승격
```

규칙:
- **일반적인지 아닌지 그 자리에서 판단하지 않는다.** 그건 승격 세션의 일.
- 고치려 들지 않는다. 기록만 한다. 작업 흐름을 끊지 않는 게 포착률의 전부.
- 설치된 스킬 파일을 직접 수정하지 않는다 — `/plugin update` 때 날아간다.
  **프로젝트는 소비자이지 편집자가 아니다.**

전역 규칙(`~/.claude/CLAUDE.md`)에도 이 포착 규약이 인라인돼 있어 어느 프로젝트에서든 발동한다.

---

## 2. 수거 (프로젝트 종료 시점)

프로젝트가 끝나면 그 프로젝트의 `skill-feedback.md`를 `_capture/`로 옮긴다.

```
_capture/YYMMDD_<프로젝트>_<스킬>.md
```

---

## 3. 승격 (프로젝트 종료 시점 — 게이트 4개)

수거된 항목마다 통과 여부를 따진다. **하나라도 못 넘으면 `_capture/`에 남겨둔다.**

> **상태는 `git fetch` 후 git·PR에서 확인한다 — 문서에 옮겨 적지 않는다.**
> 손으로 적은 현황은 즉시 낡고, 낡은 로컬 클론이 겹치면 오판정이 난다.
> (260721 실측: starfield는 이미 머지 완료였는데 로컬 클론만 보고 "PR 대기"로 판정했다.)

1. **일반성** — 이 프로젝트 특수값(경로·예산·이벤트명·고유 팔레트)이 섞였나?
   섞였으면 정본에 넣지 말고 프로젝트 config(`.claude/<skill>.local.md`)로 뺀다.
2. **재현성** — 2개 이상 프로젝트에서 관측됐나? 1곳뿐이면 대기.
   단 이 게이트는 **스킬 정본에만** 적용한다. 루프 규정 문서 자체의 결함(`[루프/*]`)은
   1곳 관측으로 고쳐도 되지만, ④삭제 우선은 똑같이 적용한다.
3. **회귀** — 기존 테스트·baseline이 떨어지지 않나? 테스트 없는 스킬은 먼저 테스트부터.
4. **삭제 우선** — 추가가 아니라 교체·삭제로 풀리나?

> **누적 ≠ 강화.** 스킬은 커질수록 좋아지지 않는다 — 컨텍스트를 먹고, 모델이 규칙을 덜 지키고,
> 느려진다. humanize는 v1.2~v1.4에서 이미 한 번 비대해져 롤백했다(`69229af`).
> 승격 세션의 질문은 "무엇을 더할까"가 아니라 **"무엇을 더하고 무엇을 뺄까"**다.

승격되면: 정본 수정 → 회귀 테스트 → 버전 bump → **PR** → 머지 → 프로젝트가 의도적으로 당김.

> **정본 변경은 PR로만.** `main` 직접 push 금지 — 스킬 강화·신규 등록·버전 bump 전부 해당.
> 이유는 리뷰가 아니라 **되돌릴 수 있는 지점**이다. 정본은 다른 프로젝트가 당겨 쓰는 물건이라,
> 잘못 들어가면 소비자 쪽에서 조용히 깨진다. PR이 회귀 테스트 결과와 "무엇을 뺐는지"를
> 기록으로 남기는 자리이기도 하다.
> 작업은 브랜치 + 격리 워크트리에서, 로컬 커밋까지. push·PR은 사람이 판단한다.

## 4. 성숙도 2단계

| 단계 | 사용처 | 방식 | 이유 |
|---|---|---|---|
| **실험** | 1곳 | 프로젝트 로컬에 두고 자유롭게 변형. 의식 없음 | 아직 의존자가 없으니 속도 우선 |
| **안정** | 2곳+ | 마켓 등록 + 포착→승격 루프 + 버전 | 의존자가 생겼으니 재현성 우선 |

**졸업 조건**: 2개 프로젝트에서 실사용 + 인터페이스 안정 + 지침 영어 작성 + 트리거 문서화.
한 번만 쓴 스킬은 승격하지 않는다 — 마켓 비대화 방지.

---

## 5. 회귀 방지 (강화를 주장이 아니라 사실로)

"좋아졌다"를 증명할 수단이 없으면 감으로 누적된다.

- 스킬마다 회귀 테스트를 **함께 배포**한다. (예: `plugins/higgsfield/hooks/gate-test.sh` — 76 assertion,
  실제로 우회 버그 2건을 잡았다. `plugins/humanize-korean/tests/run_tests.py` — 168 assertion.)
- 지표형 스킬은 baseline을 둔다. (예: `plugins/humanize-korean/` 의 `metrics.py`+`baseline.json`.)
- 승격마다 돌려서 이전 버전 대비 결과를 남긴다.

이 문서는 타임리스 규약만 담는다.*
