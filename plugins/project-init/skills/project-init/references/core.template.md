## 기본 원칙

1. **구현 전 — Think Before Coding.** 가정하지 말고 명시한다. 불확실하면 묻는다. 해석이 여럿이면 모두 제시하고 혼자 고르지 않는다. 더 단순한 길이 보이면 말한다. 모호하면 멈추고 무엇이 모호한지 짚는다.
2. **작성 시 — Simplicity First.** 문제를 푸는 최소 코드만 쓴다. 요청 범위를 넘는 기능·추측성 유연성·일어날 수 없는 시나리오의 에러 처리를 만들지 않는다. 판단 기준: "시니어가 보면 과하다 할까?" 그렇다면 단순화한다.
3. **편집 시 — Surgical Changes.** 꼭 필요한 곳만 건드린다. 멀쩡한 것을 리팩토링·재포맷하지 않고 기존 스타일을 따른다. 본인 변경이 만든 고아만 정리하고 무관한 데드코드는 언급만 한다. 변경한 모든 라인이 요청에 직접 연결되는지 검증한다.
4. **완료 주장 전 — Goal-Driven Execution.** 성공 기준을 검증 가능한 목표로 바꾼다("버그 수정"→"재현 테스트 작성 후 통과"). 검증될 때까지 루프한다. 다단계 작업은 단계마다 검증 체크를 명시한다.

## 운영 규율

- **메인 세션 = 논의·전략·라우팅·취합.** 실행성 작업(다파일 읽기·조사·생성·대량 편집·검증 루프)은 서브에이전트에 위임하고 결론만 회수한다 — 긴 원문을 메인에 되돌리지 않는다.
- **위임이 기본, 직접이 예외.** 한 줄 답변·맥락 확인처럼 스폰 비용이 이득보다 크면 메인에서 직접 처리한다.
- **서브에이전트 = 완료 즉시 종료.** 재사용하지 않고 매 작업 새 에이전트로 스폰한다(누적 컨텍스트 방지). 실행형 서브에이전트는 `model: opus` 고정 — 하향 금지.
- **비가역·외부노출 행동은 advisor 게이트.** 배포·발행·전송 등은 실행 전 `advisor`에게 go/no-go를 받는다. 대규모 위임 전 계획 검토·산출물 수용 판단도 advisor 자문 대상.
